# 上线须知 — MPC 安全修复（sign.rs 私钥泄露 + reshare 公钥绑定）

> 适用版本: 修复 `partial_s` 私钥泄露 + F-008 reshare 绑定的这批改动
> 状态: 代码已修 + 单测通过，**尚未真机/双端联调**

本批修复涉及 MPC 协议线路格式与门限安全模型,上线动作与普通后端发布不同,务必按下面两点执行。

## 1. App 与 Server 必须同版本发布(线路不兼容)

修复删除了 `SignRound2Message::MtARequest` 的 `partial_s` 字段。该枚举经 `bincode` 序列化在**设备(FFI)**与**服务器(api-server `mpc_participant`)**之间传输,两端编译同一份 `mpc-core`。

字段变更导致跨版本反序列化失败:

| 设备 App | 服务器 | 结果 |
|----------|--------|------|
| 旧(带 `partial_s`) | 新 | ❌ `bincode::deserialize` 失败,签名报错 |
| 新 | 旧 | ❌ 同上 |
| 新 | 新 | ✅ |
| 旧 | 旧 | ✅(但仍有私钥泄露漏洞) |

**要求:**
- 新服务器二进制与新 App 二进制**同时发布**,不能只更后端。
- 部署瞬间正在进行的旧会话会失败;用户重试即可(签名可重新发起,无副作用)。
- 上线前在 staging 用**真机装新 App + 连新后端**,跑一次真实签名 + 一次 reshare 冒烟,确认握手不炸。

## 2. 存量钱包需评估密钥轮换(补丁救不了已泄露的私钥)

🔴 严重项 `partial_s` 私钥泄露**在 `main` 上已存在**(非本批引入)。任何用**旧代码已签过链上交易**的钱包:

- 设备明文发送 `k_0⁻¹·m`,服务器同样知道 `m` → 可推导 `k_0⁻¹ → k_0 → k`,再结合链上公开的 `(r,s)` 解出**完整私钥 `d`**。
- 即一个诚实但好奇/被攻破的服务器,对这类钱包已可独占资产控制权。

**关键结论:补丁只堵住未来的签名,救不了已泄露的密钥。**

- **Reshare 不够。** `reshare.rs` 保留 `public_key = old_share.public_key`,即刷新分片但**不改底层私钥 `d`**。已泄露的 `d` 经 reshare 后仍是同一个 `d`,资产地址不变,风险不消。
- **必须走全新 DKG。** 对已签过名的存量钱包,应用新 DKG 生成**全新公钥(=新地址)**,再把资产从旧地址迁移过去。这是产品/安全决策,需评估:
  - 哪些钱包在旧代码期间发起过链上签名(查 `transactions` 表)。
  - 迁移方案(用户引导、gas 承担、批量 vs 自愿)。

> 本项目为 Alpha 阶段。若旧代码从未在生产环境为真实资产签过名(仅内测/演示地址),则无存量暴露,直接同版本发布即可。上线前请核实这一前提。
